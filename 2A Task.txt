## First, What Is Big-O *Actually* Saying?

Forget the textbook definition for a second.

Big-O answers one question: **"If I double the input size, what happens to my program's running time?"**

That's it. It's a measure of **growth**, not speed.

Your laptop might run a bad algorithm fast today. But put 10 lakh records in front of it — suddenly it freezes. Big-O predicts *when* your code will fail at scale.

---

## The Cheat Sheet You Need to Memorize

| Notation | Name | What it means in plain English |
|---|---|---|
| **O(1)** | Constant | Input size doesn't matter. Always same time. |
| **O(log n)** | Logarithmic | Each step cuts the problem in half. |
| **O(n)** | Linear | You touch every element once. |
| **O(n log n)** | Log-linear | Sorting territory. Efficient but not trivial. |
| **O(n²)** | Quadratic | Loop inside a loop. Danger zone. |
| **O(2ⁿ)** | Exponential | Recursion gone wrong. Interview red flag. |

Think of it as a **ladder of pain** — the lower you are, the better your code.

```
O(1) ← Best
O(log n)
O(n)
O(n log n)
O(n²)
O(2ⁿ) ← Worst
```

---

## Concrete Examples (The Ones Interviewers Love)

### O(1) — Finding an element by index
```python
arr = [10, 20, 30, 40]
print(arr[2])  # Always 1 step. Array has 4 or 4 million items — doesn't matter.
```
Accessing `arr[2]` is **always** one operation. That's O(1).

---

### O(n) — Simple linear search
```python
def find(arr, target):
    for item in arr:        # Visits every element
        if item == target:
            return True
    return False
```
Worst case? You check all *n* elements. Double the array → double the time. That's **O(n)**.

---

### O(n²) — The nested loop trap
```python
def has_duplicate(arr):
    for i in range(len(arr)):
        for j in range(len(arr)):   # ← This inner loop is the killer
            if i != j and arr[i] == arr[j]:
                return True
    return False
```
100 elements → 10,000 operations.
1,000 elements → 1,000,000 operations.

**This is what Infosys asks you to optimize.** They show you this code and ask: *"Can you do better?"* (Yes — use a HashSet. That's O(n).)

---

### O(log n) — Binary Search
```python
def binary_search(arr, target):
    low, high = 0, len(arr) - 1
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1
```
1,000 elements? At most **10 steps**. (Because 2¹⁰ = 1024.)
1,000,000 elements? At most **20 steps**.

This is *why* interviewers love binary search. It's the poster child of O(log n).

---

## The 3 Rules You Apply While Writing Code

**Rule 1 — Drop the constants.**
O(2n) is just O(n). O(100) is just O(1). Constants don't matter at scale.

**Rule 2 — Drop the non-dominant terms.**
O(n² + n) is just O(n²). The smaller term is irrelevant when n is large.

**Rule 3 — Nested loops multiply, sequential loops add.**
```python
for i in range(n):       # O(n)
    pass

for j in range(n):       # O(n)
    pass
# Total: O(n) + O(n) = O(n)  ← Still linear!

for i in range(n):
    for j in range(n):   # O(n) inside O(n)
        pass
# Total: O(n × n) = O(n²)  ← Now it's quadratic!
```

## What TCS/Infosys/Cognizant Actually Ask

> *"What is the time complexity of Bubble Sort?"* → **O(n²)**

> *"Which sorting algorithm is O(n log n)?"* → **Merge Sort, Quick Sort (avg), Heap Sort**

> *"You have an unsorted array. Find if a target exists. What's the best complexity you can achieve?"* → **O(log n) — but only after sorting it first (O(n log n)), so total is O(n log n). Or use a HashSet for O(n) build + O(1) lookup.**

> *"Optimize this nested loop"* → The answer is almost always: use a **HashMap/HashSet** to trade space for time.

---

## The One Line That Impresses Every Interviewer

When they ask you to analyze code, don't just say *"it's O(n²)"*. Say:

> *"This runs in O(n²) time due to the nested iteration. We can optimize it to O(n) by using a hash map, trading O(n) space for linear time."*

That one sentence tells them you understand **the time-space tradeoff** — and that's a senior engineer's thinking.

**Step 1 — Definition**

---

**Step 2 — Concrete Example**

Big-O measures how your algorithm's time or space grows relative to input size — O(1) stays flat, O(n) grows linearly, O(n²) explodes. The practical skill is looking at code and immediately spotting the dominant term: one loop is O(n), a loop inside a loop is O(n²), halving the problem each step is O(log n). Master this and you answer not just *"does it work?"* but *"does it work at scale?"* — which is exactly what every TCS, Infosys, and Cognizant interviewer is listening for.
